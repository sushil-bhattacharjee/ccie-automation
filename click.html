📖 Chapter 1: @click.argument vs @click.option

▼
Feature@click.argument@click.option
Required?Always requiredOptional by default
Order?Position mattersAny order (uses flags)
Passed byscript.py John--name John
Default?With default= becomes optionalSupports default=
Flags?❌ Not supported✅ is_flag=True
Good forEssential inputs (filenames)Optional behavior (switches)

Example

@click.command() @click.argument('filename') # Positional: script.py data.txt @click.option('--verbose', is_flag=True) # Named: --verbose def process(filename, verbose): pass
💡 Key Rule: Arguments for essential inputs, options for optional/adjustable behavior.

📖 Chapter 3: Boolean Flag Pair vs is_flag

▼

Boolean Flag Pair: --all/--project

The / creates two mutually exclusive flags.

@click.option('--all/--project', default=True) def cmd(all): if all: # --all was used (or default) else: # --project was used
Commandall value
cmdTrue (default)
cmd --allTrue
cmd --projectFalse

Simple Flag: is_flag=True

On/off switch. Default is always False (implicit).

@click.option('--showgid', '-g', is_flag=True) def cmd(showgid): # showgid=False when absent, True when present
⚠️ Key Difference:
• Boolean pair: Use default=True or default=False
• is_flag: Default is implicitly False (no need to write it)

📖 Chapter 4: click.Choice() Syntax

▼

click.Choice() is a class that restricts input to specific values.

@click.option('--format', type=click.Choice(['json', 'xml']), default="json")

Valid vs Invalid Syntax

SyntaxValid?Why?
click.Choice(['json', 'xml'])✅ YesPasses a list

click.Choice['json', 'xml']❌ NoWrong syntax
⚠️ Default Must Be String: Use default="json" NOT default=True

📖 Chapter 5: Short vs Long Options

▼
TypeSyntaxExample
Long optionTwo dashes ----outformat
Short optionOne dash --o

Defining Both

@click.option('--outformat', '-o', type=click.Choice(['lines', 'raw'])) # Long Short
💡 Note: Click actually accepts '--o' (double dash with one letter) too, but convention is single dash for short options.

Usage

$ python cli.py --outformat lines # Long form $ python cli.py -o lines # Short form $ python cli.py --o lines # Also works!

📖 Chapter 6: @click.group() - Creating Groups

▼

A group is a container that holds multiple commands.

@click.group() def cli(): """Main CLI application""" pass if __name__ == "__main__": cli()

Usage

$ python cli.py --help Commands: command1 command2
✅ Remember: The entry point in if __name__ is the group function, not a command.

📖 Chapter 7: @click.command() - Commands

▼

Two Ways to Register Commands

Method 1: Using @group.command() decorator

@cli.command() # Registers directly to 'cli' group def listprojects(): pass

Method 2: Using @click.command() + add_command()

@click.command() # Standalone command def listprojects(): pass cli.add_command(listprojects) # Manually add to group
⚠️ Don't Mix! If you use @group.command(), don't also call add_command() - it would register twice!

📖 Chapter 8: add_command() Method

▼

Manually adds commands or subgroups to a group.

Syntax

group.add_command(command) # Uses function name group.add_command(command, name="alias") # Custom name

Example: Adding with Custom Name

@click.command() def namespace(): # Function name: namespace pass individual.add_command(namespace, name="path") # CLI name: path # Usage: cli individual path

What Can Be Added?

What You're AddingMethod
Command to groupgroup.add_command(cmd)
Subgroup to groupgroup.add_command(subgroup)
💡 Note: There is NO add_subcommand() method. Everything uses add_command().

📖 Chapter 9: Context: @click.pass_context & @click.pass_obj

▼

Context allows sharing data between group and commands.

@click.pass_context

Used on the group to initialize shared data.

@click.group() @click.pass_context def gitlab(ctx): ctx.ensure_object(dict) config = json.load(open("pat.json")) ctx.obj["api"] = GitlabAPI(config["PAT"])

@click.pass_obj

Used on commands to receive the shared data.

@gitlab.command() @click.pass_obj def listprojects(obj): # obj = ctx.obj projects = obj["api"].get("projects")
DecoratorUsed OnParameter
@click.pass_contextGroupctx (full context)
@click.pass_objCommandsobj (ctx.obj only)

📖 Chapter 10: Nested Subgroups

▼

Groups can contain other groups, creating nested CLI structures.

# Main group @click.group() def gitlab(): pass # Sub-group @click.group() def individual(): pass # Commands for sub-group @click.command() def namespace(): pass @click.command() def commitmessage(): pass # Wire it all together gitlab.add_command(individual) individual.add_command(namespace, name="path") individual.add_command(commitmessage, name="message")

Resulting Structure

gitlab ├── listprojects ├── project ├── projectpath └── individual # Sub-group ├── path # Renamed from 'namespace' └── message # Renamed from 'commitmessage'

Usage

$ python cli.py individual --help $ python cli.py individual path --help $ python cli.py individual message

Question 1 MCQ
What is the correct decorator for the following requirement?
📋 Expected Help Output:
Options: --all / --project Show all projects or specific project [default: (all: show all projects)]

Rule: --all/--project is a boolean pair, default is --all

@click.option('--all/--project', default="all", show_default="all: show all projects")
@click.option('--all/--project', default=True, help="Show all projects or specific project", show_default="all: show all projects")
@click.option('--all/--project', is_flag=True, default="all")
@click.option('--all', '--project', default=True, help="Show all projects")
💡 Explanation

✅ Correct: @click.option('--all/--project', default=True, help="...", show_default="...")

  • Boolean pairs use / between flags
  • default=True makes --all the default
  • help= provides description text
  • show_default= shows custom default text

❌ Why others are wrong:

  • default="all" — Boolean pairs need True/False, NOT strings
  • is_flag=True — Not used with boolean pairs; is_flag is for simple on/off flags
  • '--all', '--project' — Comma creates two separate options, NOT a boolean pair. Must use /
Question 2 MCQ
Which is the correct way to define a Choice option with "lines" as the default?
📋 Expected Help Output:
Options: --outformat, -o [lines|raw] [default: lines]
@click.option('--outformat', '-o', type=click.Choice('lines', 'raw'), default="lines")
@click.option('--outformat', '-o', type=click.Choice['lines', 'raw'], default=True)
@click.option('--outformat', '-o', type=click.Choice(['lines', 'raw']), default="lines", show_default=True)
@click.option('--outformat', '-o', choices=['lines', 'raw'], default="lines")
💡 Explanation

✅ Correct: type=click.Choice(['lines', 'raw']), default="lines", show_default=True

  • click.Choice() requires a list as argument
  • default="lines" — Default must be a string matching a choice
  • '-o' — Short option uses single dash

❌ Why others are wrong:

  • click.Choice('lines', 'raw') — Passes 2 separate arguments instead of 1 list → TypeError
  • click.Choice['lines', 'raw'] — Square brackets are for indexing, not calling classes → TypeError
  • choices=['lines', 'raw'] — No choices= parameter in Click; must use type=click.Choice([...])
  • default=True — Choice options need string default, not boolean
Question 3 MCQ
What decorator combination is needed for the main group to share data with commands?
@click.group() and @click.pass_obj on the group
@click.group() + @click.pass_context on group, @click.pass_obj on commands
@click.command() + @click.pass_context on everything
@click.share_data() decorator
💡 Explanation

✅ Correct: @click.group() + @click.pass_context on group, @click.pass_obj on commands

  • Group uses @click.pass_context to access ctx and initialize ctx.obj
  • Commands use @click.pass_obj to receive ctx.obj as obj parameter

❌ Why others are wrong:

  • @click.pass_obj on group — pass_obj receives data, it doesn't create/initialize it
  • @click.command() on everything — Groups need @click.group(), not @click.command()
  • @click.share_data() — This decorator doesn't exist in Click

Question 5 MCQ
What is the default value of is_flag=True when the flag is NOT used?
None
True
False (implicit, no need to write it)
"" (empty string)
💡 Explanation

✅ Correct: False (implicit, no need to write it)

  • Flag absent → False
  • Flag present → True

  • None — Flags are boolean, never None
  • True — Would be inverted logic (rare, requires explicit default=True)
  • "" — Flags are boolean, never strings
