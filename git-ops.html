git revertSafe

Creates a NEW commit that undoes changes from a previous commit. History is preserved.

# Undo the last commit
git revert HEAD

# Undo a specific commit
git revert abc1234

# Result: Original commit stays, new "revert commit" added
$ git log --oneline
6ededa0 Revert "Third commit"   ← NEW commit
decfe02 Third commit            ← Original preserved
💡 Use When: Undoing changes on shared branches (main, develop). Safe because it doesn't rewrite history.

git resetRewrites History

Moves HEAD backward. Three modes with different behaviors:

FlagCommitsFile ContentChanges State
--softRemovedUnchangedStaged ✓
--mixedRemovedUnchangedUnstaged
--hardRemovedReverted!Discarded ⚠️
 
# Reword last commit message (--soft)
git reset --soft HEAD~1
git commit -m "Better message"

# Squash last 4 commits into one
git reset --soft HEAD~4
git commit -m "Combined feature"

# Completely discard last commit (DESTRUCTIVE!)
git reset --hard HEAD~1
⚠️ Warning: --hard permanently discards changes! Never use reset on shared branches.

git rebaseRewrites History

Replays commits on top of another branch. Creates linear history but destroys original commits.

# Update feature branch with latest main
git checkout feature
git rebase main
# Your feature commits are RECREATED on top of main

# Interactive rebase to squash/edit commits
git rebase -i HEAD~4
# Opens editor: change 'pick' to 'squash' or 'edit'

git cherry-pickSafe

COPIES a specific commit to your current branch. Original stays in source branch.

# Copy one commit from feature to main
git checkout main
git cherry-pick abc1234

# Copy multiple commits
git cherry-pick abc1234 def5678

# Result: NEW commits with DIFFERENT hashes
💡 Key Point: Cherry-pick COPIES (not moves). Original commit remains. New commit gets a different hash.

git format-patch / git amSafe

Export commits as patch files for transfer (email, USB, air-gapped networks).

# Create patch files for last 2 commits
git format-patch -2
# Creates: 0001-First-change.patch, 0002-Second-change.patch

# Apply patch (changes only, no commit)
git apply 0001-First-change.patch

# Apply patch AND create commit with original metadata
git am 0001-First-change.patch

git bisectSafe

Binary search to find the commit that introduced a bug. Halves search space each step.

# Start bisect session
git bisect start
git bisect bad              # Current commit is broken
git bisect good abc1234     # This commit was working

# Git checks out middle commit - test it
git bisect good   # or 'bad' based on test result
# Repeat until culprit found

git bisect reset            # Return to original state

# Automated bisect with script
git bisect run ./test-script.sh
# Exit 0 = good, Exit 1 = bad
💡 Efficiency: For 50 commits, bisect needs only ~6 tests (log₂50), not 50!

📍 HEAD~N Notation

f277ad0 (HEAD)  ← HEAD~0 (current)
5b75696         ← HEAD~1 (1 back)
6ededa0         ← HEAD~2 (2 back)
decfe02         ← HEAD~3 (3 back)

# Examples:
git reset HEAD~1     # Go back 1 commit
git rebase -i HEAD~4 # Interactive rebase last 4
git diff HEAD~2 HEAD # Diff between 2 commits ago and now
