Git Workflows Explained

A comprehensive guide to Git branching workflows for CCIE DevNet Expert certification preparation. Includes visual diagrams, merge commands, and best practices.

📊 Quick Comparison

Workflow Branches Complexity Best For
Gitflow main, develop, feature/*, release/*, hotfix/* ● High Scheduled releases
GitHub Flow main, feature/* ● Low Continuous deployment
GitLab Flow main, staging, production ● Medium Environment promotion
Trunk-Based main only ● Lowest Mature CI/CD teams
Forking fork + upstream ● Medium Open source

1. Gitflow Complex

The most structured workflow with dedicated branches for features, releases, and hotfixes. Ideal for teams with scheduled release cycles.

Branch Structure

main         ← Production-ready code (long-lived)
  │
  └── develop    ← Integration branch (long-lived)
        │
        ├── feature/*  ← New features (short-lived)
        │
        ├── release/*  ← Release preparation (short-lived)
        │
        └── hotfix/*   ← Emergency fixes (short-lived)

Visual Flow

main:       ●─────────────────────●─────────●
                                  ↑         ↑
release:                 ●───●          │
                        /    \         │
develop:  ●────●────●────●──────●─────────●
               ↑    ↑          ↑
feature:       ●────●          │
                               │
hotfix:                             ●────●
                                    (from main)

Merge Rules

Branch Created FROM Merges TO
feature/* develop develop
release/* develop main + develop
hotfix/* main main + develop

Commands

# Feature workflow
git checkout develop
git checkout -b feature/snmp-monitoring
# ... work on feature ...
git checkout develop
git merge feature/snmp-monitoring
git branch -d feature/snmp-monitoring

# Release workflow
git checkout develop
git checkout -b release/v1.0
# ... final testing, version bump ...
git checkout main
git merge release/v1.0
git tag -a v1.0 -m "Release v1.0"
git checkout develop
git merge release/v1.0

# Hotfix workflow (production is down!)
git checkout main
git checkout -b hotfix/critical-bug
# ... fix the bug ...
git checkout main
git merge hotfix/critical-bug
git checkout develop
git merge hotfix/critical-bug

2. GitHub Flow Simple

A lightweight workflow with only main and feature branches. Perfect for continuous deployment and small teams.

Visual Flow

main:      ●────●────●────●────●────●
                \        /    \    /
feature:        ●──────●      ●──●
                    ↓           ↓
              Pull Request  Pull Request

Commands

# Everything branches from main
git checkout main
git checkout -b feature/new-api

# Work on feature
git commit -am "Add new API endpoint"
git push origin feature/new-api

# Create Pull Request on GitHub
# After review, merge via PR

git checkout main
git pull origin main
git branch -d feature/new-api

Key Points

Long-lived branchesmain only
Feature branches frommain
Merge viaPull Request
Deploy frequencyAfter every merge

3. GitLab Flow Medium

An environment-based workflow that promotes code through staging environments before production.

Visual Flow

main:        ●────●────●────●
                       │
staging:           └────●────●
                            │
production:              └────●

Commands

# Feature merged to main
git checkout main
git merge feature/api

# Promote to staging
git checkout staging
git merge main

# After testing, promote to production
git checkout production
git merge staging

4. Trunk-Based Development Simplest

Everyone commits directly to main. Requires feature flags to hide incomplete work and mature CI/CD.

Visual Flow

main:  ●──●──●──●──●──●──●──●──●
        ↑  ↑  ↑  ↑  ↑  ↑  ↑  ↑
       All developers commit directly
       (use feature flags to hide incomplete work)

Feature Flags Example

# Feature flag to hide incomplete work
FEATURE_FLAGS = {
    "new_checkout": False,  # Hidden from users
    "dark_mode": True,      # Visible to users
}

if FEATURE_FLAGS["new_checkout"]:
    show_new_checkout()     # Still in development
else:
    show_old_checkout()     # Production-ready

Key Points

Branchesmain only (or very short-lived)
Feature flags✓ Essential
CI/CD✓ Required
Best forMature teams, Google/Facebook style

5. Forking Workflow Medium

Contributors fork the repository and submit Pull Requests to upstream. Standard for open source projects.

Visual Flow

upstream/main:     ●────●────●────●
                                   ↑
                              Pull Request
                                   │
your-fork/main:    ●────●────●
                         ↑
your-fork/feature: ●────●

Commands

# Fork on GitHub, then clone your fork
git clone https://github.com/YOUR-USER/repo.git

# Add upstream remote
git remote add upstream https://github.com/ORIGINAL/repo.git

# Create feature branch
git checkout -b feature/fix-bug
# ... work ...
git push origin feature/fix-bug

# Create Pull Request to upstream on GitHub

# Keep your fork updated
git fetch upstream
git checkout main
git merge upstream/main

6. Centralized Workflow Simplest

Everyone commits directly to main with no branches. Similar to SVN. Best for small teams or learning Git.

Visual Flow

main:  ●──●──●──●──●──●──●──●
        ↑  ↑  ↑  ↑  ↑  ↑  ↑  ↑
      Alice Bob Carol Alice Bob Carol

      No branches, no PRs, no reviews

Commands

# Everyone works on main
git clone https://github.com/company/project.git

# Make changes and push
git commit -am "Add feature"
git push origin main

# If someone else pushed first
git pull origin main   # Merge their changes
git push origin main   # Now push yours

🔧 Git Merge Strategies

Strategies with Rename Detection
Strategy Rename Support Use Case
ort ✓ Yes Default strategy for 2 branches
recursive ✓ Yes Synonym for ort (since v2.50)
subtree ✓ Yes Merge into subdirectory
resolve ✗ No Simple 3-way merge
octopus ✗ No Merge 3+ branches
ours ✗ No Ignore other branch entirely

Strategy Commands

# Default merge (uses ort)
git merge feature-branch

# Explicitly specify strategy
git merge -s ort feature-branch
git merge -s octopus branch1 branch2 branch3

# Strategy OPTIONS (-X) vs Strategy (-s)
git merge -X ours feature    # Option: favor our changes in conflicts
git merge -s ours feature    # Strategy: ignore other branch entirely!

# Rename detection threshold
git merge -X find-renames=50 feature

Rename Files Correctly

# Correct command (git rename does NOT exist!)
git mv old_name.py new_name.py
git commit -m "Rename old_name to new_name"

# View history including before rename
git log --follow new_name.py
