3. Authentication Quick Reference

NX-API REST (DME) — Token-based

# Step 1: Login (-d implies POST)
export token=$(curl -sk "https://$NXOS_URI/api/aaaLogin.json" \
  -H 'content-type: application/json' \
  -d '{
        "aaaUser" : {
            "attributes" : {
                "name" : "expert",
                "pwd" : "1234QWer!"
            }
        }
}' | jq -r '.imdata[0].aaaLogin.attributes.token')

# Step 2: Use token
curl -sk "https://$NXOS_URI/api/class/l2BD.json" \
  -H "Cookie: APIC-cookie=$token"

RESTCONF & NX-API CLI — Basic Auth

# Using -u (curl encodes for you)
curl -sk "https://192.168.89.73/restconf/data/Cisco-NX-OS-device:System" \
  -u expert:'1234QWer!'

# Using Authorization header (you encode manually)
echo -n 'expert:1234QWer!' | base64
# ZXhwZXJ0OjEyMzRRV2VyIQ==

curl -sk "https://192.168.89.73/restconf/data/Cisco-NX-OS-device:System" \
  -H 'Authorization: Basic ZXhwZXJ0OjEyMzRRV2VyIQ=='
Both -u user:pass and -H 'Authorization: Basic ' are identical.
-u can go anywhere in the curl command — position doesn't matter.

9. DME Filter Syntax Reference

 
# query-target-filter — filters the queried objects (class queries)
/api/class/l2BD.json?query-target-filter=eq(l2BD.id,"2110")

# rsp-subtree-filter — filters children (MO queries with subtree)
/api/mo/sys/bd.json?rsp-subtree=children&rsp-subtree-class=l2BD&rsp-subtree-filter=eq(l2BD.fabEncap,"vlan-2110")&rsp-subtree-include=required

# wcard — wildcard match
?query-target-filter=wcard(l2BD.name,"Challenge")

# eq — exact match
?query-target-filter=eq(l2BD.id,"2110")

Filter gotchas

- Double quotes around string values — single quotes cause invalid responses

- No space after comma — eq(l2BD.id,"2110") not eq(l2BD.id, "2110")

- Escape quotes in bash — \"2110\" inside double-quoted URLs

- query-target-filter filters the class; rsp-subtree-filter filters children

- rsp-subtree-include=required filters parent objects based on matching children

10. NX-API CLI Response Structure & jq Gotchas

NX-API CLI always wraps arrays in TABLE_* / ROW_* pairs:

{
  "ins_api": {
    "outputs": {
      "output": {
        "body": {
          "TABLE_intf": {
            "ROW_intf": [ ... ]
          }
        }
      }
    }
  }
}

jq gotchas

- Hyphenated keys need bracket syntax: .["intf-name"] not .intf-name

- Leading dot required: .prefix not prefix

- No trailing comma before }

- Single result = object, multiple results = array — watch for inconsistency

- Shorthand {foo} only works for simple keys, hyphenated keys need {name: .["intf-name"]}

- Array iterator [] must be outside quotes: ."BD-list"[] not ."BD-list[]"

13. Server vs Client Side Filtering

NX-OS Server vs Client Side Filtering

Overview

NX-OS offers two main approaches to filter API query results. DME supports rich server-side filtering via query parameters. RESTCONF has limited server-side options, so complex filtering happens client-side with tools like jq.

Server-side Filtering (DME)

Filtering happens on the switch — only matching objects are returned.

Comparison Operators

OperatorMeaningExample
eqequaleq(l1PhysIf.mtu,"9216")
nenot equalne(l1PhysIf.adminSt,"up")
ge>=ge(l1PhysIf.mtu,"9000")
le<=le(l1PhysIf.mtu,"1500")
gt>gt(l1PhysIf.mtu,"1500")
lt<lt(l1PhysIf.mtu,"9216")
wcardwildcardwcard(l1PhysIf.id,"eth1")

Logical Operators

# AND — all conditions must match
?query-target-filter=and(eq(l1PhysIf.adminSt,"up"),ge(l1PhysIf.mtu,"9000"))

# OR — any condition matches
?query-target-filter=or(eq(l2BD.name,"prod99"),eq(l2BD.name,"dev99"))

Filter Parameter Types

ParameterPurposeUsed With
query-target-filterFilters the queried class objects directly/api/class/ queries
rsp-subtree-filterFilters children within a subtree response/api/mo/ with rsp-subtree=children
rsp-subtree-filter only makes sense when paired with rsp-subtree=children or rsp-subtree=full.

Examples

# Simple: Find VLAN 2110
curl -sk "https://$NXOS_URI/api/class/l2BD.json?query-target-filter=eq(l2BD.id,\"2110\")" \
  -H "Cookie: APIC-cookie=$token" \
  | jq '.imdata[].l2BD.attributes | {id, name, fabEncap}'

# Medium: Interfaces with MTU >= 9000
curl -sk "https://$NXOS_URI/api/class/l1PhysIf.json?query-target-filter=ge(l1PhysIf.mtu,\"9000\")" \
  -H "Cookie: APIC-cookie=$token" \
  | jq '.imdata[].l1PhysIf.attributes | {id, mtu}'

# Complex: Interfaces up + up + MTU >= 9000
curl -sk "https://$NXOS_URI/api/class/l1PhysIf.json?query-target-filter=and(eq(l1PhysIf.adminSt,\"up\"),eq(l1PhysIf.operSt,\"up\"),ge(l1PhysIf.mtu,\"9000\"))" \
  -H "Cookie: APIC-cookie=$token" \
  | jq '.imdata[].l1PhysIf.attributes | {id, adminSt, operSt, mtu}'

# Subtree: All VLANs under bd entity
curl -sk --globoff "https://$NXOS_URI/api/mo/sys/bd.json?rsp-subtree=children&rsp-subtree-class=l2BD&rsp-subtree-filter=wcard(l2BD.fabEncap,\"vlan\")&rsp-subtree-include=required" \
  -H "Cookie: APIC-cookie=$token" \
  | jq '.imdata[].bdEntity.children[].l2BD.attributes | {id, name, fabEncap}'

Filter Gotchas (DME)

- Double quotes around string values — single quotes cause invalid responses

- No space after comma — eq(l2BD.id,"2110") not eq(l2BD.id, "2110")

- Escape quotes in bash — \"2110\" inside double-quoted URLs

- URL-encode if special characters cause issues — %2C for comma, %22 for quotes

Client-side Filtering (RESTCONF + jq)

Filtering happens in your terminal after fetching all data.

RESTCONF Query Parameters (limited)

ParameterPurposeExample
contentConfig vs operational data?content=config
depthLimit subtree depth?depth=2
fieldsSelect specific fields?fields=id;name
RESTCONF does NOT support eq(), ge(), wcard(), and(), or() or any DME-style filter operators.

jq select() Syntax

# Basic comparison
select(.mtu >= 9000)

# Equality
select(.adminSt == "up")

# AND — combine with 'and'
select(.adminSt == "up" and .operSt == "up")

# OR — combine with 'or'
select(.name == "prod99" or .name == "dev99")

# NOT
select(.adminSt != "down")

# Complex
select(.adminSt == "up" and .operSt == "up" and .mtu >= 9000)

Examples

# Simple: All interfaces with MTU >= 9000
curl -sk "https://$NXOS_URI/restconf/data/Cisco-NX-OS-device:System/intf-items/phys-items" \
  -H 'accept: application/yang-data+json' \
  -u $NXOS_USERNAME:$NXOS_PASSWORD \
  | jq '."phys-items"."PhysIf-list"[] | select(.mtu >= 9000) | {id, mtu}'

# Complex: Interfaces up + up + MTU >= 9000
curl -sk "https://$NXOS_URI/restconf/data/Cisco-NX-OS-device:System/intf-items/phys-items" \
  -H 'accept: application/yang-data+json' \
  -u $NXOS_USERNAME:$NXOS_PASSWORD \
  | jq '."phys-items"."PhysIf-list"[] | select(.adminSt == "up" and .operSt == "up" and .mtu >= 9000) | {id, adminSt, operSt, mtu}'

# Filter VLANs by name containing "prod"
curl -sk "https://$NXOS_URI/restconf/data/Cisco-NX-OS-device:System/bd-items" \
  -H 'accept: application/yang-data+json' \
  -u $NXOS_USERNAME:$NXOS_PASSWORD \
  | jq '."bd-items"."bd-items"."BD-list"[] | select(.name | test("prod")) | {id, name, fabEncap}'

jq Gotchas

- Hyphenated keys need bracket syntax: .["intf-name"] not .intf-name

- Leading dot required: .prefix not prefix

- select() string comparison uses == (double equals)

- select() numeric comparison uses >=, <=, >, < — but the value must be a number, not a string

- test() for regex/substring match — equivalent of DME's wcard()

Side-by-Side Comparison

Query: Find physical interfaces with operSt=up, adminSt=up, mtu >= 9000

DME (server-side): 
curl -sk "https://$NXOS_URI/api/class/l1PhysIf.json?query-target-filter=and(eq(l1PhysIf.adminSt,\"up\"),eq(l1PhysIf.operSt,\"up\"),ge(l1PhysIf.mtu,\"9000\"))" \
  -H "Cookie: APIC-cookie=$token" \
  | jq '.imdata[].l1PhysIf.attributes | {id, adminSt, operSt, mtu}'
RESTCONF (client-side): 
curl -sk "https://$NXOS_URI/restconf/data/Cisco-NX-OS-device:System/intf-items/phys-items" \
  -H 'accept: application/yang-data+json' \
  -u $NXOS_USERNAME:$NXOS_PASSWORD \
  | jq '."phys-items"."PhysIf-list"[] | select(.adminSt == "up" and .operSt == "up" and .mtu >= 9000) | {id, adminSt, operSt, mtu}'

14. RESTCONF CRUD Operations Reference

RESTCONF CRUD — reference guide

RFC 8040 · NX-OS 9.3 · URI rules, payload wrapping, error conditions

Mental model GET DELETE PUT POST PATCH Wrapping rules Error conditions Method comparison NETCONF mapping Exam gotchas

The mental model — it's a file system

Think of the YANG tree as a folder structure. The URI is the path, the payload is the file content, and the HTTP method is the action.

/System / bd-items / bd-items / BD-list=vlan-100 ↑ ↑ ↑ ↑ root folder folder THE FILE (target) C: Users Documents report.docx
One simple rule: The URI says WHERE. The payload says WHAT. The method says HOW.
Method URI ends at... Payload contains... ───────── ────────────────────────── ────────────────────────── GET THE FILE (target) nothing DELETE THE FILE (target) nothing PUT THE FILE (target) just the content (bare) POST THE FOLDER (parent) file name + content (wrapped) PATCH ANY LEVEL (target→root) fills the gap down to the file

GET — read a resource

GET
"Show me what's at this path"
NETCONF equivalent: <get-config> / <get>
RuleDetail
URI points toTHE TARGET — the object you want to read
PayloadNONE — never send a body with GET
Response200 OK + JSON/XML body with resource data
GET /restconf/data/.../bd-items/bd-items/BD-list=vlan-100 ↑ URI = what you want to read payload = NONE

What can you GET?

# Specific VLAN GET .../bd-items/bd-items/BD-list=vlan-100 → returns vlan-100 # All VLANs GET .../bd-items/bd-items → returns all BD-list entries # Specific interface GET .../phys-items/PhysIf-list=eth1%2F2 → returns eth1/2 # Entire system GET .../Cisco-NX-OS-device:System → returns everything (huge!)
Error: If you send a payload with GET, the server ignores it — but it's technically a protocol violation. Never include -d with GET requests.

DELETE — remove a resource

DELETE
"Remove what's at this path"
NETCONF equivalent: edit-config (operation="delete")
RuleDetail
URI points toTHE TARGET — the object you want to remove
PayloadNONE — DELETE never accepts a body
Response204 No Content (success) or 404 if resource doesn't exist
Multi-resource?NO — one DELETE = one target. Need 2 deletes? Need 2 requests.
DELETE /restconf/data/.../bd-items/bd-items/BD-list=vlan-100 ↑ URI = what gets deleted payload = NONE
Danger zones:
DELETE .../bd-items → deletes ALL VLANs, not just one! DELETE .../System → deletes ENTIRE system config! DELETE .../BD-list=vlan-100 → deletes only vlan-100 ✓
Cannot batch deletes. Unlike PATCH which can modify multiple resources at a common ancestor, DELETE has no payload — the URI IS the instruction. Two targets = two requests, always.

PUT — create or replace a resource

PUT
"Place this thing exactly here"
NETCONF equivalent: edit-config (operation="create/replace")
RuleDetail
URI points toTHE TARGET ITSELF — the exact object
PayloadBARE ATTRIBUTES — no wrapping (URI already identifies the object)
If existsREPLACES ENTIRELY — unspecified fields reset to defaults!
If doesn't existCREATES it
PUT /restconf/data/.../bd-items/bd-items/BD-list=vlan-100 ↑ URI = THE OBJECT itself payload = just attributes, NO wrapping payload: { "fabEncap": "vlan-100", ← bare attributes "name": "MyVlan" ← no "BD-list" wrapper needed }
PUT replaces entirely. If eth1/2 has mtu=1500, accessVlan=vlan-100, speed=10G and you PUT with only {"id":"eth1/2","mtu":9216}, the result is: mtu=9216, accessVlan=DEFAULT, speed=DEFAULT. Everything not in the payload resets. Use PATCH if you only want to change one field.

Correct vs wrong payloads

Correct — bare attributes

{ "fabEncap": "vlan-100", "name": "MyVlan" }

Wrong — wrapped with BD-list

{ "BD-list": [ {"fabEncap":"vlan-100"} ] } → double path error

POST — create a new resource

POST
"Create a new child under this parent"
NETCONF equivalent: edit-config (operation="create")
RuleDetail
URI points toTHE PARENT — one level above the target
PayloadWRAPPED — must include child list/container name
If exists409 CONFLICT — POST is strictly create-only
If doesn't existCREATES it
Multi-resource?NO — POST goes to ONE parent only
POST /restconf/data/.../bd-items/bd-items ← URI = PARENT ↑ one level ABOVE the target payload: { "BD-list": [ ← WRAPPER tells server what to create { "fabEncap": "vlan-300", "name": "MyVlan" } ] }
POST = create-only. If the resource already exists → 409 Conflict. This is the #1 exam trap. Physical interfaces (eth1/X) ALWAYS exist on the switch — you can NEVER POST to create them. Use PATCH or PUT instead.

Why wrapping is mandatory for POST

The URI only says "go to bd-items" — it doesn't say WHAT to create. The wrapper ("BD-list":[...]) tells the server "create a BD-list entry with these attributes". Without it, the server has no idea what child type to create.

POST restrictions

POST .../BD-list=vlan-300 → ERROR: POST can't target the object itself POST .../System → ERROR: POST must go to the DIRECT parent not an ancestor (that's PATCH territory) POST .../bd-items/bd-items → CORRECT: direct parent, wrapped payload

What can and can't be POSTed

ResourcePOST works?Why
vlan-200 (doesn't exist)YesVLANs are created/deleted dynamically
loopback99 (doesn't exist)YesLogical interfaces can be created
eth1/12 (physical port)409Physical ports always exist — can only modify
vlan-400 (already exists)409POST = create-only, resource exists

PATCH — merge/update a resource

PATCH
"Merge this data into whatever I point at"
NETCONF equivalent: edit-config (operation="merge")
RuleDetail
URI points toANY LEVEL — target, parent, grandparent, or root
PayloadFILLS THE GAP — from URI level down to the actual data
If existsMERGES — only specified fields change, others untouched
If doesn't existCREATES it (NETCONF merge semantics)
Multi-resource?YES — target a common ancestor, include all branches
PATCH is the safest and most flexible method. It's the ONLY method that can: create what's missing, update what exists, and handle multiple resources in one request. This is NOT standard HTTP PATCH behaviour — it's RESTCONF-specific (RFC 8040 merge semantics).

PATCH at different URI levels — same result

# Level 1: URI = target → payload = bare attributes PATCH .../bd-items/bd-items/BD-list=vlan-100 {"fabEncap":"vlan-100","name":"MyVlan"} # Level 2: URI = parent → payload wraps with child name PATCH .../bd-items/bd-items {"BD-list":[{"fabEncap":"vlan-100","name":"MyVlan"}]} # Level 3: URI = grandparent → payload wraps two levels PATCH .../bd-items {"bd-items":{"BD-list":[{"fabEncap":"vlan-100","name":"MyVlan"}]}} # Level 4: URI = root → payload wraps full tree PATCH .../Cisco-NX-OS-device:System {"bd-items":{"bd-items":{"BD-list":[{"fabEncap":"vlan-100","name":"MyVlan"}]}}}

Wrapping rules — when and why

The rule

Whatever the URI doesn't cover, the payload must fill in. The deeper the URI goes, the less wrapping the payload needs.

Visual: same VLAN, different URI depths

YANG tree: System / bd-items / bd-items / BD-list=vlan-100 L4 L3 L2 L1 (target) URI at L1 (target): payload = bare attributes URI at L2 (parent): payload = 1 level wrapping URI at L3 (grandpa): payload = 2 levels wrapping URI at L4 (root): payload = 3 levels wrapping (full tree)
URI levelMethodPayload
L1: BD-list=vlan-100PUT / PATCH{"fabEncap":"vlan-100","name":"X"}
L2: bd-items (parent)POST / PATCH{"BD-list":[{"fabEncap":"vlan-100","name":"X"}]}
L3: bd-items (grandpa)PATCH only{"bd-items":{"BD-list":[...]}}
L4: System (root)PATCH only{"bd-items":{"bd-items":{"BD-list":[...]}}}

Method determines which levels are allowed

MethodAllowed URI levelWrapping required?
GETTarget (L1)No payload
DELETETarget (L1)No payload
PUTTarget only (L1)Never — bare attributes
POSTDirect parent only (L2)Always — one level wrapping
PATCHAny level (L1–L4)Depends — fills the gap from URI to target

Common wrapping mistakes

PUT .../BD-list=vlan-100 + {"BD-list":[...]} → double path: URI says BD-list, payload says BD-list again POST .../bd-items + {"fabEncap":"vlan-100"} → no wrapper: server doesn't know which child to create PATCH .../PhysIf-list=eth1%2F11 + {"PhysIf-list":[...]} → double path: URI already covers PhysIf-list POST .../System + {"bd-items":{"bd-items":{"BD-list":[...]}}} → POST must go to DIRECT parent, not an ancestor PATCH .../System + {"bd-items":{"bd-items":{"BD-list":[...]}}} → correct: PATCH can target any ancestor with full tree wrapping

ErrorWhen it happensExample

Error decision tree

POST fails? ├── Resource exists? → 409 Conflict ├── URI targets the object? → Error (must target parent) ├── URI targets an ancestor? → Error (must target DIRECT parent) └── Payload not wrapped? → Error (server doesn't know what to create) PUT/PATCH fails? ├── Payload wraps a level the URI covers? → unknown-element ├── Slash not encoded in URI? → 404 Not Found ├── Wrong content-type? → 400 Bad Request └── Wrong auth method? → 401 Unauthorized DELETE fails? ├── Payload included? → Invalid (DELETE has no body) ├── Resource doesn't exist? → 404 Not Found └── URI too broad? → Deletes more than intended!

NETCONF mapping — why RESTCONF methods behave differently

RESTCONF methods aren't standard HTTP — they map to NETCONF operations. This is why PATCH creates missing resources (merge), and POST fails on existing ones (create).

RESTCONFNETCONF operationBehaviour
GET<get-config> / <get>Read configuration/operational data
POSTedit-config (operation="create")Create only — fails if exists
PUTedit-config (operation="create/replace")Creates new or replaces entirely
PATCHedit-config (operation="merge")Creates if missing, updates if exists
DELETEedit-config (operation="delete")Removes — fails if doesn't exist
For the exam: whenever you see RESTCONF, think NETCONF operations — not standard HTTP behaviour. PATCH = merge (creates + updates). POST = create (fails if exists). PUT = create/replace (overwrites everything).
